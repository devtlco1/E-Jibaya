# شرح خطة حذف البيانات والنسخ الاحتياطي

## ١) حذف البيانات (قبل ١٧ فبراير)

### المطلوب (كما فهمته):
- **تُحذف:** كل سجل أُضيف أو عُدّل أو تغيّر بتاريخ **١٧ فبراير أو قبله**
- **تبقى:** كل سجل أُضيف أو عُدّل أو تغيّر **بعد ١٧ فبراير**

### التاريخ المستخدم:
- **١٧ فبراير ٢٠٢٥** (أو ٢٠٢٦ حسب السنة الحالية)
- المعيار: `created_at` و/أو `updated_at` أو `submitted_at` حسب الجدول

### الجداول المتأثرة وترتيب الحذف (بسبب العلاقات):

| الترتيب | الجدول | عمود التاريخ | ملاحظات |
|--------|--------|--------------|---------|
| 1 | `record_changes_log` | تاريخ التغيير | سجلات التعديلات |
| 2 | `record_photos` | `created_at` | صور مرتبطة بالسجلات |
| 3 | `record_locations` | `created_at` | مواقع مرتبطة بالسجلات |
| 4 | `activity_logs` | `created_at` | سجلات النشاط |
| 5 | `collection_records` | `submitted_at` و `updated_at` | السجلات الرئيسية |
| 6 | `user_sessions` | `created_at` أو `expires_at` | جلسات المستخدمين |
| 7 | `backup_logs` | `created_at` | سجلات النسخ الاحتياطي |
| 8 | `branch_manager_employees` | `created_at` | ربط مدير الفرع بالموظفين |
| 9 | `branch_manager_field_agents` | `created_at` | ربط مدير الفرع بالمحصلين |
| 10 | `users` | `created_at` و `updated_at` | **حذر:** قد نحذف حسابات نظام |

### تحذير بخصوص جدول `users`:
- حذف المستخدمين قد يمنع تسجيل الدخول إذا حُذف حسابك.
- **الخيار الأفضل:** عدم حذف `users` إلا إذا طلبت ذلك صراحة، أو حذف المستخدمين الذين `created_at <= 2025-02-17` فقط (مع التأكد أن حسابك أحدث).

### منطق الحذف:
```sql
-- مثال لـ collection_records:
DELETE FROM collection_records 
WHERE (submitted_at AT TIME ZONE 'Asia/Baghdad')::date <= '2025-02-17'
   OR (updated_at AT TIME ZONE 'Asia/Baghdad')::date <= '2025-02-17';
```
- السجلات التي `submitted_at` أو `updated_at` بعد ١٧ فبراير **تبقى**.

## ٢) النسخ الاحتياطي (Backup)

### المطلوب:
- مجلد باسم `backup` (أو `باك اب`)
- بداخله نسخة من ملفات النظام

### ما سيُنسخ:
- `src/` — الكود المصدري
- `public/` — الأصول الثابتة
- `supabase/migrations/` — ملفات الهجرة
- `index.html`, `package.json`, `vite.config.ts`, `tailwind.config.js`, `tsconfig.json`, إلخ
- **لا يُنسخ:** `node_modules`, `dist`, `.env`, `env.production`, ملفات البيانات الكبيرة (مثل `invest exal.xlsx`)

### هيكل المجلد:
```
backup/
├── backup_2025-02-26/   (أو التاريخ الحالي)
│   ├── src/
│   ├── public/
│   ├── supabase/
│   ├── index.html
│   ├── package.json
│   └── ...
└── README.txt   (شرح محتويات النسخة)
```

---

## ٣) ترتيب التنفيذ المقترح

1. **أولاً:** إنشاء النسخ الاحتياطي (مجلد `backup`)
2. **ثانياً:** كتابة سكربت SQL للحذف (migration أو سكربت منفصل)
3. **ثالثاً:** تنفيذ الحذف يدوياً عبر Supabase SQL Editor أو سكربت Node (بعد تأكيدك)

---

## ٤) ما أحتاج تأكيده منك

1. **السنة:** ٢٠٢٥ أم ٢٠٢٦؟
2. **جدول users:** هل نحذف المستخدمين الذين أُضيفوا قبل ١٧ فبراير؟ (قد يؤثر على تسجيل الدخول)
3. **جدول backup_info:** هل نحذف سجلات النسخ الاحتياطي القديمة أيضاً؟
4. **التوقيت:** هل نستخدم توقيت بغداد `Asia/Baghdad` أم UTC؟

بعد تأكيدك، أُنفّذ النسخ الاحتياطي أولاً، ثم أُعدّ سكربت الحذف جاهزاً للتنفيذ.
